# Сравнение загрузки зависимостей: ExtJS 7.9 vs React

Этот документ предназначен для разработчиков, которые переходят с ExtJS 7.9 на React и хотят понять, как меняется подход к описанию и загрузке зависимостей.

## Архитектурная разница подходов

- **ExtJS 7.9 (classic)**:
  - предоставляет собственную класс‑систему (`Ext.Class`) и автозагрузчик (`Ext.Loader`, `Ext.require`);
  - зависимости объявляются в конфиге класса строками:
    - `extend: 'Ext.grid.Panel'`
    - `requires: ['MyApp.store.Users', 'Ext.window.Window']`
  - при объявлении класса (`Ext.define`) класс‑система:
    - обрабатывает `extend` через препроцессор, настраивает наследование;
    - собирает список `requires` и вызывает загрузчик для нужных классов **до** создания текущего класса.

- **React (актуальная версия)**:
  - не имеет собственной класс‑системы и автозагрузчика модулей;
  - использует стандартные ES‑модули:
    - `import React from 'react';`
    - `import Button from './Button';`
    - `const LazyPage = React.lazy(() => import('./LazyPage'));`
  - загрузкой модулей, построением графа зависимостей и разбивкой кода на чанки управляет **бандлер** (Vite, Webpack, Rollup и др.), а не сам React.

Итог: ExtJS централизует классы и загрузку в самом фреймворке, React опирается на стандарты JavaScript и экосистему инструментов.

## Как ExtJS загружает зависимости

В ExtJS:

- `extend`:
  - указывает родительский класс;
  - участвует в наследовании и в логике препроцессора `extend`;
  - при отсутствии родителя используется базовый `Ext.Base`.
- `requires`:
  - перечисляет имена классов, которые должны быть загружены заранее;
  - при создании класса вызывается `Ext.require(requires, callback)`;
  - загрузчик по строковым именам находит и подгружает JS‑файлы, затем продолжает обработку класса.

Дополнительно:

- в профилях/контроллерах фреймворк может сам вычислять массив `requires` на основе `models`, `stores`, `views` и т.д.;
- вся цепочка зависимостей классов жёстко привязана к именам пространств имён (`MyApp.view.UserGrid` и т.п.).

## Как React и бандлер загружают зависимости

В React:

- зависимости описываются через `import` и `export`:
  - статический импорт: `import Child from './Child';`
  - динамический импорт: `const Lazy = React.lazy(() => import('./Lazy'));`
- бандлер:
  - строит граф модулей на основе импортов;
  - решает, какие модули включить в стартовый бандл, а какие вынести в отдельные чанки;
  - при динамическом `import()` подгружает соответствующий чанк вместе со всеми его статическими зависимостями.

Важно:

- **React сам по себе код не грузит и не ищет файлы по строковым именам классов**;
- `React.lazy` лишь говорит: «этот компонент нужно получить из промиса модуля», а всё остальное делает инфраструктура модулей и бандлера.

## Есть ли эквивалент `requires` и `extend` в React?

- **Наследование (`extend`)**:
  - в React/TypeScript используется обычный синтаксис JS:
    - `class MyComponent extends React.Component { ... }`
  - но в современных приложениях упор делается не на наследование, а на композицию:
    - функциональные компоненты + хуки.
  - React не добавляет поверх этого своих конфигов наследования.

- **Зависимости (`requires`)**:
  - прямого аналога нет;
  - роль декларации зависимостей выполняют **ES‑импорты**;
  - статический анализ импортов даёт:
    - tree‑shaking и удаление неиспользуемого кода;
    - предсказуемый code splitting;
    - хорошую интеграцию с TypeScript, линтерами и IDE.

Таким образом, при переходе с ExtJS не нужно искать в React поле вида `requires`: его роль выполняют обычные импорты, а загрузкой управляет бандлер.

## Практические советы при миграции с ExtJS на React

- **Вместо конфигов классов — модули и компоненты**:
  - каждый ExtJS‑класс, как правило, превращается в React‑компонент или модуль;
  - зависимости классов из `requires` становятся `import`‑выражениями.

- **Вместо ExtJS‑загрузчика — настройка сборки**:
  - продумывайте структуру каталогов и модулей так, чтобы бандлеру было удобно делать code splitting;
  - используйте `React.lazy`/`Suspense` и динамический `import()` в тех местах, где раньше полагались на позднюю загрузку частей приложения.

- **Ленивая загрузка**:
  - в ExtJS загрузчик решал, какие классы подтянуть по `requires`;
  - в React вы явно указываете точки ленивой загрузки (`React.lazy`, роут‑сплиттинг, ленивые тяжёлые библиотеки), а бандлер формирует соответствующие чанки.

- **Отказ от «магии строковых имён»**:
  - вместо `'MyApp.view.UserGrid'` → `import UserGrid from './UserGrid';`;
  - это уменьшает количество ошибок из‑за опечаток и упрощает рефакторинг.

## Когда в React может понадобиться «менеджер зависимостей»

Иногда при миграции хочется воспроизвести ощущение централизованного списка зависимостей. В React‑мире это может выглядеть как:

- реестр ленивых компонентов/виджетов (как `ComponentLoader` в микропроекте `react-lazy`);
- таблица соответствий «имя → функция `() => import(...)`»;
- доп. логика: логирование загрузок, prefetch при определённых сценариях, feature‑flags.

Важно: такой менеджер остаётся **пользовательским уровнем**, а не частью React как фреймворка. Он не заменяет и не дублирует работу бандлера, а лишь организует ваш код вокруг стандартных `import()` и `React.lazy`.
