# Кастомный менеджер компонентов (ComponentLoader)

Этот документ описывает архитектурный подход, при котором ленивая загрузка компонентов инкапсулируется в специальном объекте‑менеджере (например, классе `ComponentLoader`), а не размазывается по всему коду. Такой менеджер **не является аналогом ExtJS‑загрузчика с `requires`/`extend`**: он не анализирует конфигурации классов, а просто оборачивает конкретные функции динамического импорта (`import()`) и результат `React.lazy`.

## Цели и мотивация

- **Централизация знаний** о том, какие ленивые компоненты существуют в приложении.
- **Единая точка расширения** для дополнительной логики:
  - логирование загрузок и ошибок;
  - искусственная задержка (для демонстрации/тестирования);
  - настройки повторных попыток;
  - prefetch/preload.
- **Инкапсуляция связки** «имя компонента → динамический импорт → `React.lazy`».

В микропроекте `react-lazy` это реализовано через класс `ComponentLoader` с синглтоном (`getInstance()`), который хранит:

- карту функций‑загрузчиков,
- кэш уже созданных ленивых компонентов.

## Типичный интерфейс менеджера

На уровне типов (упрощённый пример):

```ts
type ComponentName = 'Component1' | 'Component2';

type LazyLoader = () => Promise<unknown>;

interface IComponentLoader {
  loadComponent(name: ComponentName): React.LazyExoticComponent<React.ComponentType<any>>;
  isLoaded(name: ComponentName): boolean;
  getComponent(name: ComponentName): React.LazyExoticComponent<React.ComponentType<any>> | null;
}
```

Реализация в стиле OOP:

- приватное хранилище `componentMap: Record<ComponentName, LazyLoader>`;
- приватный кэш `loadedComponents: Map<ComponentName, LoadableComponent>`;
- публичные методы для управления загрузкой и доступа к компонентам.

## Как менеджер работает с зависимостями

Важно понимать, что менеджер **не управляет зависимостями внутри модулей** напрямую. Он:

- знает только функцию `loader`, которая вызывает `import('...')`;
- делегирует работу по построению графа зависимостей и разбивке на чанки бандлеру;
- на уровне React‑кода оперирует уже готовым `LazyExoticComponent`.

При вызове `loadComponent('Component1')`:

1. Менеджер находит соответствующий `loader` в `componentMap`.
2. Оборачивает его в `React.lazy(loader)` при первом обращении.
3. Кладёт результат в кэш и возвращает его вызывающему коду.

Дальше при рендере в `Suspense`:

- запускается динамический `import()`;
- подгружается чанк с модулем компонента и **всеми его зависимостями**;
- после этого React рендерит компонент как обычный.

Задача менеджера — **управлять жизненным циклом ленивых компонентов на уровне приложения**, а не ручным управлением зависимостями чанков.

## Плюсы и минусы подхода

**Плюсы:**

- Чёткая точка расширения для любой дополнительной логики вокруг ленивой загрузки.
- Возможность внедрять кросс‑срезочные аспекты:
  - метрики (время загрузки, число ошибок);
  - трейсинг;
  - политика кеширования и инвалидации.
- Удобно интегрировать с DI/Service‑слоем и тестами.

**Минусы:**

- Дополнительная абстракция, требующая поддержки.
- Возможен «бог‑объект», если в менеджер свалить слишком много ответственности.
- Требуется аккуратное версионирование ключей (`Component1`, `Component2` и т.п.).

## Рекомендации по использованию

- Использовать менеджер, когда:
  - ленивых компонентов много;
  - требуется централизованный контроль;
  - есть нефункциональные требования (метрики, логирование, fault‑tolerance).
- Для небольших приложений достаточно прямого использования `React.lazy` без отдельного слоя.
- Держать интерфейс менеджера строгим (строгие типы имён, возвращаемых компонентов) и минимальным.
