# Prefetch и Preload

Этот документ описывает механизмы **ранней подгрузки ресурсов** (чанков с кодом, шрифтов, данных и т.д.), которые дополняют ленивую загрузку компонентов.

## Интуитивное различие

- **Ленивая загрузка**: «загрузи это только тогда, когда реально понадобится».
- **Prefetch**: «скорее всего скоро понадобится — можно загрузить в фоне, когда есть свободные ресурсы».
- **Preload**: «понадобится очень скоро — загрузи как можно раньше, почти как критичный ресурс».

Эти механизмы позволяют:

- сгладить задержки при первом открытии ленивых экранов;
- более тонко управлять сетью и временем рендера.

## Prefetch

`prefetch` обычно означает:

- низкий приоритет загрузки;
- загрузка в момент простоя, когда сетевой канал и CPU не перегружены;
- ресурс может и не быть использован сразу после загрузки.

Примеры использования:

- HTML‑тег:

```html
<link rel="prefetch" href="/static/chunk-users-page.js" as="script" />
```

- API бандлера/фреймворка:
  - специальные комментарии в импортах (в зависимости от инструмента);
  - интеграция с роутером (prefetch при hover на ссылку и т.п.).

## Preload

`preload` сигнализирует браузеру:

- ресурс критичен для ближайшего рендера;
- его нужно начать загружать как можно раньше;
- но он может быть использован не сразу на момент парсинга HTML.

Пример:

```html
<link rel="preload" href="/static/chunk-above-the-fold.js" as="script" />
```

Осторожность:

- неправильное использование `preload` может привести к конкуренции с действительно критичными ресурсами и ухудшить TTFB/TTI.

## В связке с ленивой загрузкой компонентов

Практические сценарии:

- **prefetch ленивых страниц**:
  - при hover на ссылку или при видимости ссылки в viewport;
  - при простое приложения (`requestIdleCallback`) инициировать `import()` нужных модулей.
- **preload критичных виджетов**:
  - если точно известно, что пользователь почти наверняка откроет следующий экран;
  - аккуратно использовать `preload` для чанков, отвечающих за «первый экран» после редиректа/логина.

Интеграция с `ComponentLoader`:

- добавить методы вроде `prefetchComponent(name: string)`:
  - внутри они могут вызывать `componentMap[name]()` без немедленного рендера;
  - результат кешируется, и последующий `loadComponent` станет практически мгновенным.

## Риски и баланс

- Избыточный `prefetch`/`preload` может:
  - увеличить потребление трафика,
  - привести к загрузке кода, который не будет использован.
- Недостаточная подгрузка заранее — к заметным «ступенькам» при открытии новых экранов.

Рекомендации:

- начинать с простой ленивой загрузки;
- добавлять `prefetch`/`preload` только на основе реальных метрик и наблюдений;
- тестировать поведение на разных сетях и устройствах.
