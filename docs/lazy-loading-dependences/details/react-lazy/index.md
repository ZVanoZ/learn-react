# React.lazy и Suspense (детально)

Этот документ предназначен для разработчиков, которые хотят глубоко понимать, как устроена ленивая загрузка компонентов с помощью `React.lazy`, как она связана с динамическим импортом (`import()`) и работой бандлера.

## Концепция и API

- `React.lazy(loader)` принимает функцию-загрузчик `loader: () => Promise<Module>`, которая возвращает промис модуля (результат динамического `import()`).
- Модуль должен экспортировать компонент по умолчанию (`export default`), который будет отрендерен после загрузки.
- Возвращаемое значение — `React.LazyExoticComponent<T>` (в типах `LoadableComponent`), который можно использовать как обычный React‑компонент, но его обязательно нужно оборачивать в `Suspense`.

Типы (упрощённо):

```ts
declare function lazy<T extends React.ComponentType<any>>(
  loader: () => Promise<{ default: T }>
): React.LazyExoticComponent<T>;
```

## Динамический импорт и чанки

Ключевая техническая деталь: `React.lazy` сам по себе **не загружает код**. Он использует функцию‑загрузчик, которая обычно реализуется через динамический импорт:

```ts
const HeavyComponent = React.lazy(() => import('./HeavyComponent'));
```

При сборке бандлер:

- видит выражение `import('./HeavyComponent')`;
- выносит код модуля `./HeavyComponent` и всех его **статических зависимостей** (другие компоненты, функции, импортируемые библиотеки) в отдельный чанк;
- в основном бандле оставляет только «указатель» на этот чанк (идентификатор и логику подгрузки).

Таким образом, **при первом рендере ленивого компонента**:

1. React вызывает `loader()` (функцию в `React.lazy`).
2. `loader()` выполняет `import('./HeavyComponent')`, что запускает загрузку чанка через механизм бандлера.
3. Пока промис не завершился, `Suspense` показывает `fallback`.
4. После загрузки модуля React получает `default`‑экспорт и рендерит его как обычный компонент.

Важно: **цепочка зависимостей** (`HeavyComponent` → дочерние компоненты → утилиты → библиотеки) входит в тот же или в связанные чанки в зависимости от стратегии бандлера. Разработчик на уровне `React.lazy` обычно не управляет этим напрямую.

## Пример из микропроекта `react-lazy`

В микропроекте `react-lazy` ленивая загрузка реализована через класс‑менеджер `ComponentLoader`:

- `ComponentLoader` хранит карту `componentMap: Record<string, () => Promise<any>>`, где для каждого имени компонента определена функция‑загрузчик.
- Внутри этой карты используются динамические импорты:
  - `import('../components/Component1/Component1')`
  - `import('../components/Component2/Component2')`
- Каждый загрузчик обёрнут в дополнительный `Promise` с `setTimeout`, чтобы продемонстрировать поведение при «долгой» загрузке.
- Метод `loadComponent(name)` оборачивает соответствующую функцию в `React.lazy` и кэширует результат.

Вместе с типами:

- `LoadableComponent = React.LazyExoticComponent<React.ComponentType<any>>;`
- `ComponentInstance` хранит:
  - идентификатор,
  - тип (`'Component1' | 'Component2'`),
  - собственно ленивый компонент.

Рендер:

- `ComponentContainer` оборачивает каждый `LoadableComponent` в `Suspense` с текстовым `fallback`.
- `ErrorBoundary` защищает от ошибок при загрузке и рендеринге.

## Поведение зависимостей компонентов

При использовании `React.lazy` и динамического `import()`:

- все **статические импорты** внутри лениво загружаемого модуля (например, `import Child from './Child';`, `import { helper } from '../utils';`, `import Chart from 'some-chart-lib';`) становятся частью графа зависимостей для этого чанка;
- бандлер сам решает, включать ли их:
  - в тот же чанк (по умолчанию для локальных зависимостей),
  - или выносить в отдельные чанки (например, для крупных внешних библиотек с повторным использованием).

С точки зрения React‑кода:

- **все необходимые зависимости подгружаются автоматически** при первом `import()` ленивого модуля;
- не нужно вручную заботиться о загрузке дочерних компонентов — достаточно корректно использовать модули и импорты.

Особые случаи:

- динамические импорты внутри ленивого модуля (`import('./SubPart')` по условию) могут создавать дополнительные уровни ленивой загрузки;
- если одна и та же тяжёлая библиотека используется в нескольких ленивых компонентах, бандлер может вынести её в отдельный общий чанк (strategy‑зависимо).

## Ограничения и подводные камни

1. **Поддержка только default‑экспорта**
   - Модуль должен экспортировать компонент через `export default`.
   - Иначе придётся писать адаптер: `() => import('./Comp').then(mod => ({ default: mod.NamedExport }))`.

2. **Обязательное использование Suspense**
   - Без `Suspense` React не знает, как отобразить состояние загрузки.
   - Нельзя рендерить `React.lazy`‑компонент вне дерева, где наверху есть `Suspense`.

3. **SSR и гидратация**
   - Базовая связка `React.lazy` + `Suspense` рассчитана на клиентский рендеринг.
   - При SSR нужны дополнительные инструменты (например, библиотеки, которые умеют собирать информацию о использованных чанках и подключать их на клиенте до гидратации).

4. **Гранулярность чанков**
   - Слишком мелкая разбивка (много маленьких ленивых модулей) может привести к росту количества сетевых запросов.
   - Слишком крупные чанки снижают пользу ленивой загрузки.
   - Оптимальный размер зависит от структуры приложения и поведения пользователей.

## Паттерны и best practices

- **Фича‑модули**: объединяйте компоненты и их утилиты по фичам (модулям) и загружайте их лениво как единое целое.
- **Route‑контейнеры**: лениво грузите страницы и их зависимости через `React.lazy` в связке с роутером.
- **Отложенные тяжёлые виджеты**: изолируйте графики, редакторы и другие тяжёлые компоненты в отдельные ленивые модули.
- **Явные границы ответственности**: используйте отдельные файлы/каталоги для ленивых фич, чтобы бандлеру было проще выделять чанки.
