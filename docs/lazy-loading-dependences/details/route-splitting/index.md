# Route‑based code splitting

Этот документ описывает подход, при котором код страниц и их зависимостей загружается лениво в зависимости от маршрута (URL). Это один из самых распространённых вариантов ленивой загрузки в React‑приложениях.

## Основная идея

- Каждому маршруту (странице) соответствует отдельный ленивый компонент.
- Компонент страницы загружается только тогда, когда пользователь открывает соответствующий путь.
- Бандлер выносит код страницы и её зависимостей в отдельный чанк.

Таким образом:

- стартовый бандл содержит только код роутера, оболочки приложения и наиболее часто используемых частей;
- редкие или тяжёлые страницы подгружаются по требованию.

## Базовый пример с React Router

Упрощённый пример (концептуально, без привязки к версии роутера):

```tsx
const UsersPage = React.lazy(() => import('./pages/UsersPage'));
const ReportsPage = React.lazy(() => import('./pages/ReportsPage'));

function AppRouter() {
  return (
    <React.Suspense fallback={<div>Загрузка страницы...</div>}>
      <Routes>
        <Route path="/users" element={<UsersPage />} />
        <Route path="/reports" element={<ReportsPage />} />
      </Routes>
    </React.Suspense>
  );
}
```

Бандлер:

- создаёт отдельные чанки для `UsersPage` и `ReportsPage` и их зависимостей;
- по `import()` подгружает соответствующий чанк при первом заходе пользователя на маршрут.

## Работа с зависимостями страниц

Когда вы лениво загружаете компонент страницы:

- все её **статические импорты** (дочерние компоненты, хелперы, сервисы и т.д.) попадают в тот же или связанные чанки;
- крупные общие библиотеки (например, графики) могут быть вынесены в отдельный shared‑чанк.

Это означает, что:

- обычно нет необходимости вручную управлять зависимостями страниц — достаточно правильно структурировать код по фичам;
- важно следить за тем, чтобы общие, тяжёлые зависимости не подтягивались в стартовый бандл без необходимости.

## Типовые паттерны

- **Фича‑модули по маршрутам**:
  - `/dashboard` → модуль `dashboard` (страница, виджеты, сервисы);
  - `/admin` → модуль `admin` и так далее.
- **Комбинирование с менеджером компонентов**:
  - сами страницы грузятся лениво через роутер,
  - внутри страниц используются дополнительные ленивые виджеты/компоненты, управляемые через `ComponentLoader`.

## Плюсы и минусы

**Плюсы:**

- естественное разбиение кода по функциональным областям (маршрутам);
- пользователи не скачивают код страниц, которые никогда не посещают;
- хорошо масштабируется для крупных SPA.

**Минусы и нюансы:**

- требуется аккуратная работа с `fallback`‑состояниями (скелетоны, лоадеры);
- слишком мелкие страницы могут привести к избыточному количеству запросов;
- нужно следить за тем, чтобы критичные зависимости не были случайно вынесены в ленивые чанки.
