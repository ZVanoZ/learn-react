# SSR и ленивая загрузка

Этот документ даёт обзор того, как ленивая загрузка компонентов и зависимостей взаимодействует с серверным рендерингом (SSR) и гидратацией.

## Проблема согласования чанков

При SSR:

- React генерирует HTML на сервере;
- на клиенте React должна «подхватить» этот HTML и продолжить работу (гидратация);
- если на клиенте не загружены нужные чанки с кодом, возможны ошибки и несоответствие разметки.

Ленивая загрузка усложняет ситуацию:

- сервер может отрендерить часть дерева, используя информацию о чанках;
- клиент должен знать, какие чанки были использованы, и загрузить их **до** начала гидратации.

## Ограничения базового React.lazy

Стандартная связка:

- `React.lazy`
- `React.Suspense`

изначально была спроектирована для клиентского рендеринга. Для полноценного SSR с код‑сплиттингом исторически использовались дополнительные библиотеки и решения, которые:

- отслеживают, какие динамические импорты были выполнены на сервере;
- формируют манифест с перечнем чанков для включения в HTML;
- обеспечивают корректный порядок загрузки скриптов на клиенте.

## Подходы к SSR с ленивой загрузкой

На архитектурном уровне существуют варианты:

- **Специализированные библиотеки и фреймворки**:
  - Next.js и аналоги берут на себя синхронизацию чанков между сервером и клиентом;
  - продвинутые решения умеют стримить HTML и подмешивать данные о чанках.
- **Сторонние решения поверх React**:
  - библиотеки для SSR‑code‑splitting с собственными компонентами (аналог `lazy`/`Suspense`, но с поддержкой SSR).

Общая идея:

- при рендеринге на сервере собирается список использованных ленивых модулей;
- в HTML встраиваются теги `<script>`/`<link>` для соответствующих чанков;
- на клиенте эти чанки загружаются до начала или во время гидратации.

## Практическая перспектива

Для микропроектов уровня `react-lazy`:

- обычно достаточно понимать, что базовый `React.lazy` ориентирован на клиент;
- для серьёзного SSR лучше опираться на фреймворк (Next.js и др.), который:
  - уже решает задачи по согласованию чанков;
  - даёт готовые абстракции для ленивой загрузки.

При проектировании архитектуры важно:

- заранее решить, планируется ли SSR;
- если да — выбирать механизмы ленивой загрузки и code splitting, которые поддерживаются выбранным SSR‑фреймворком.
