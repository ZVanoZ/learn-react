## Ленивая загрузка зависимостей React‑компонент

Этот документ предназначен **для начинающих разработчиков** на React. Здесь мы простыми словами разберём, что такое ленивая загрузка компонентов и их зависимостей, зачем она нужна и какие механизмы существуют. Для каждого механизма есть отдельный подробный документ для более опытных разработчиков.

### Зачем нужна ленивая загрузка

- **Уменьшить размер первого бандла**: при первой загрузке приложения браузер скачивает только минимально необходимый код.
- **Ускорить первый рендер**: пользователь быстрее видит рабочий интерфейс.
- **Отложить загрузку редких экранов и тяжёлых библиотек**: графики, редакторы, карты и т.п. можно подгрузить только тогда, когда они действительно понадобятся.

Современные инструменты сборки (Webpack, Vite, Rollup и др.) позволяют автоматически разбивать код на части (чанки) и подгружать их по требованию. React добавляет к этому удобные механизмы на уровне компонентов.

## Список доступных механизмов

Ниже перечислены основные подходы к ленивой загрузке компонентов и их зависимостей. Для каждого подхода сначала даётся **упрощённое объяснение**, а затем ссылка на **детальный документ для профессионалов**.

### React.lazy и Suspense

**Идея:** вы описываете компонент так, чтобы он загружался не сразу, а только когда его нужно показать. React сам подгружает код через динамический импорт и показывает запасной контент (заглушку), пока идёт загрузка.

**Простой пример:**

```tsx
// Компонент будет загружен только при первом рендере MyPage
const MyHeavyWidget = React.lazy(() => import('./MyHeavyWidget'));

function MyPage() {
  return (
    <React.Suspense fallback={<div>Загрузка...</div>}>
      <MyHeavyWidget />
    </React.Suspense>
  );
}
```

**Что важно помнить новичку:**

- ленивый компонент всегда нужно оборачивать в `React.Suspense`;
- пока компонент загружается, пользователь видит `fallback` (заглушку);
- все зависимости этого компонента (дочерние компоненты, функции и библиотеки, которые он импортирует) загружаются вместе с ним в том же чанке.

### Кастомный менеджер компонентов (ComponentLoader)

**Идея:** вместо того чтобы напрямую вызывать `React.lazy` в каждом месте, вы создаёте отдельный объект‑менеджер. Он:

- знает, как по имени компонента найти нужный модуль;
- управляет загрузкой и кэшированием ленивых компонентов;
- может добавлять дополнительную логику (например, искусственную задержку, логирование, политику повторных попыток).

В микропроекте `react-lazy` используется такой подход: есть класс `ComponentLoader`, который внутри использует `React.lazy(() => import(...))` и карту соответствий «имя компонента → функция динамического импорта».

**Что важно помнить новичку:**

- это более «архитектурный» уровень — надстройка над `React.lazy`;
- зависимости компонент по‑прежнему загружаются обычным способом через динамический импорт и бандлер;
- менеджер просто помогает централизованно управлять тем, какие ленивые компоненты есть в приложении.

[Кастомный менеджер компонентов — детально для профессионалов](details/dependency-manager/index.md)

### Ленивая загрузка через роутинг (route‑based code splitting)

**Идея:** многие экраны приложения соответствуют маршрутам (URL). Можно загружать код экрана только тогда, когда пользователь открывает соответствующий маршрут, а не заранее.

На практике это означает, что:

- компоненты страниц описываются как ленивые;
- роутер (например, React Router) показывает заглушку, пока грузится код страницы;
- бандлер выносит код каждой страницы и её зависимостей в отдельный чанк.

**Что важно помнить новичку:**

- это удобный способ разбивать приложение по экранам;
- зависимости конкретной страницы (её вложенные компоненты и библиотеки) обычно попадают в тот же чанк, что и страница;
- реализация зависит от используемого роутера, но идея одна и та же.

[Route‑based code splitting — детально для профессионалов](details/route-splitting/index.md)

### Ленивая загрузка тяжёлых зависимостей (библиотек)

Иногда проблема не в самих React‑компонентах, а в больших библиотеках, которые они используют: графики, редакторы, карты, сложные UI‑наборы.

**Идея:** такие библиотеки можно загружать только там, где они реально нужны, и не включать их код в основной бандл.

Принцип похож на `React.lazy`, только акцент на **отделении тяжёлой зависимости** и загрузке её по требованию.

**Что важно помнить новичку:**

- если библиотека нужна только на одной странице, её можно подгружать лениво вместе с этой страницей;
- если она нужна редко, это хороший кандидат на вынесение в отдельный чанк;
- важно не переусердствовать: слишком много маленьких чанков может замедлить приложение из‑за большого числа сетевых запросов.

[Ленивая загрузка тяжёлых зависимостей — детально для профессионалов](details/heavy-deps/index.md)

### Prefetch и Preload (ранняя подгрузка зависимостей)

Ленивая загрузка откладывает загрузку до первого обращения, но иногда полезно **подготовиться заранее**: начать скачивать код в фоне ещё до того, как пользователь откроет нужный экран.

**Идея:** браузеру или бандлеру даётся подсказка: «этот ресурс скоро понадобится, можно загрузить его заранее, но не блокировать текущий рендер».

**Что важно помнить новичку:**

- `prefetch` подходит для ресурсов, которые «возможно скоро пригодятся»;
- `preload` — для ресурсов, которые **точно понадобятся очень скоро**;
- эти механизмы помогают сгладить задержку при открытии следующего экрана, но увеличивают общий объём загружаемых данных.

[Prefetch и Preload — детально для профессионалов](details/prefetch-preload/index.md)

### SSR и ленивая загрузка

При серверном рендеринге (SSR) часть React‑кода выполняется на сервере. Ленивая загрузка в этом случае усложняется: нужно согласовать, какие чанки были использованы на сервере, и убедиться, что клиент может их подхватить.

**Что важно помнить новичку:**

- базовый `React.lazy` и `Suspense` изначально ориентированы на клиентский рендеринг;
- для продвинутого SSR обычно используются дополнительные библиотеки и настройки сборки;
- эта тема важна для крупных проектов, но для первых шагов с ленивой загрузкой достаточно понимать общую идею.

[SSR и ленивая загрузка — детально для профессионалов](details/ssr/index.md)

## Как это связано с микропроектом `react-lazy`

Микропроект `react-lazy` демонстрирует один из возможных подходов:

- компоненты (`Component1`, `Component2`) загружаются лениво с помощью `React.lazy` и динамического `import()`;
- используется класс `ComponentLoader`, который управляет соответствием «имя компонента → ленивый компонент» и кэширует результаты загрузки;
- контейнер `ComponentContainer` оборачивает каждый лениво загружаемый компонент в `Suspense` и показывает текст «Загрузка компонента...», пока код не подгружен;
- `ErrorBoundary` защищает интерфейс от ошибок, которые могут произойти при загрузке или рендеринге компонента.

Важно понимать, что **зависимости компонентов** (их дочерние компоненты и импортируемые модули) в этом подходе подгружаются автоматически вместе с чанком, который создаёт бандлер для динамического `import()`. Специальной дополнительной логики для вложенных зависимостей в `react-lazy` нет — ею занимается инструмент сборки.

Подробнее о том, как именно это устроено внутри, и какие есть альтернативы, можно прочитать в детальных документах из раздела `details/`.
